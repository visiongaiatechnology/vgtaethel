# VGT AETHEL — Beta V2 Release Audit

Audit date: 2026-07-14  
Target release identity: `1.0.0-beta.2` / `BETA V2`

## Decision

**GO for a reviewed Beta V2 source release candidate; NO-GO for an unsigned public Windows binary.** The application compiles, its local automated gates are green, the reachable Go vulnerability scan is clean, and the native Windows runtime package starts. Public binaries still require the external P0 evidence and release-chain work below.

## P0 — required before publishing Beta V2

### 1. Real provider and orchestrator E2E evidence

The current live tests are skipped without credentials. More importantly, `chat_agent_integration_test.go` currently proves only run creation/start and explicitly does not invoke the live model loop.

Required release matrix with real accounts:

1. ordinary conversation without tools;
2. domain-model answer handed to a different Orchestrator model;
3. native tool call and verified result;
4. Writer mutation visible in Sphere;
5. Global Watch context + region + time-window visible in the UI;
6. approval request, reject, approve, pause, resume;
7. application restart with run recovery and explicit resume;
8. final report bound to verified effects;
9. empty answer, malformed tool arguments, provider 400/401/403/429/5xx, timeout and fallback;
10. no `[[THINKING]]`, tool delta, usage metadata, or generic fake-success text in user-facing reports.

At minimum this must run against Groq GPT-OSS 20B/120B and Qwen 3.6. Each additionally configured provider needs one live chat and one tool-call adapter test.

### 2. Release identity changed atomically — complete

The source now uses `1.0.0-beta.2` / `BETA V2` across `system/product.go`, Wails metadata, shell, splash/disclosure, installer defaults, release workflow and release notes. An automated consistency test guards the identity chain. The remaining external release action is the verified tag:

- backend product version;
- Wails metadata;
- title, loader badge, disclosure and sidebar badge;
- installer version and filename;
- release notes, compatibility matrix and update manifest;
- Git tag `aethel-v1.0.0-beta.2`.

### 3. Create a real Git release chain from the authoritative tree

The root `.git` directory is empty, so there is no locally provable commit, branch, remote, or tag. The clean `aethel-github` staging copy is not authoritative: it differs from the active implementation in hundreds of files and lacks the new Orchestrator.

Required:

- initialize or clone the actual GitHub repository;
- import the authoritative root tree without runtime data;
- review the complete diff;
- commit in coherent release commits;
- push a release candidate branch;
- let CI build from that exact commit;
- tag only the verified commit.

### 4. Prevent local data leakage

The development workspace contains local sessions, screenshots, logs, configuration, encrypted stores and key material. The root `.gitignore` now excludes these paths, but the first real Git import must prove they are absent from both index and history.

Required gates:

- `git ls-files` contains no `vgt_workspace`, `_logs`, screenshots, `.key`, `.pfx`, `.env`, chat/session or audit data;
- secret scanner on the full Git history;
- source archive generated by Git, never by zipping the current working directory.

### 5. Signed and reproducible Windows distribution

The current local EXE is not Authenticode-signed. Tagged releases must not proceed without certificate secrets. The workflow now fails tagged releases when signing material is absent and assembles the required native DLLs.

Still required:

- code-signing certificate and GitHub secrets;
- timestamped signature verification for EXE and installer;
- execute the hardened workflow on the release candidate;
- verify portable ZIP and installer on a clean Windows VM;
- verify uninstall/update behavior and persistence of operator data;
- publish SHA-256 checksums, build metadata and dependency manifest.

The installer compiler was not installed in the audit environment, so the final Inno Setup artifact remains unverified locally.

### 6. Clean-install voice/runtime contract

The native DLL omission is fixed in the build pipeline. Offline Sherpa model data is approximately 138 MB and is not currently included in the installer. A clean installation therefore needs one explicit product decision:

- ship a license-reviewed offline voice pack; or
- provide an authenticated/checksummed first-run voice-pack installer; or
- hide unavailable Sherpa voices and present SAPI/browser fallback as the active capability.

No unavailable voice may be shown as operational.

### 7. Full WebView UI smoke test

Static tests do not prove clickability, layout, focus, overlays or recovery behavior. Run an explicit clean-profile matrix at 100%, 125% and 150% Windows scaling and at common 16:9/16:10 resolutions.

Mandatory paths: onboarding, every navigation entry, model and Orchestrator dropdowns, provider health, chat, Persona save, Agent Tracker, approval popup from every view, Run Center, Personal Core, Nexus, Sphere window resize/minimize/maximize, Writer, Browser, weather/market widgets, Global Watch, briefing/reader, Cases/Evidence, Security/Audit, Settings, voice call and complete offline startup.

## P1 — strongly recommended for Beta V2 quality

- Finish localization beyond the current 38 tagged shell strings. Dynamic dialogs, toasts, settings, Sphere, Case and Run Center content still contain hard-coded German/English strings.
- Add a Global Watch performance budget and automated long-run soak: idle/interaction FPS, WebView GPU, heap growth, timer/rAF count, 2k/10k event datasets and visibility-background behavior.
- Move runtime state from CWD-relative `./vgt_workspace` paths to a versioned per-user application-data root with tested migration from Beta V1.5.
- Replace the lexical TF-IDF/Jaccard memory core with a local hybrid lexical + embedding index, including encrypted metadata and deterministic fallback when the embedding model is unavailable.
- Add OS-level process isolation for host-control skills. Policy/approval gates are strong application controls but execution still runs with the operator's Windows token.
- Remove the remaining CSP `unsafe-inline` dependency by extracting inline script/style and adopting a nonce/hash policy.
- Add schema-versioned migrations and rollback fixtures for every persisted store, not only selected session/key formats.
- Add crash-report transport only after an explicit endpoint/privacy decision; keep it opt-in and content-minimized.
- Configure and publish the signed update manifest/public key.
